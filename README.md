This repository is focused on Terraform modules for **Kubernetes API objects** —
plus the **Istio**, **GKE** and **Gateway API** custom resources that usually
sit alongside them. Each object type is its own module so you can compose exactly
what you need, **without** coupling your manifests to a specific cloud provider or
a monolithic "do everything" chart.

## Overview

- **What it is** — one module per Kubernetes / Istio / GKE / Gateway-API object
  kind. Standard Kubernetes objects use the native `hashicorp/kubernetes`
  provider resources; CRD-backed objects (Istio, GKE, Gateway API) are rendered
  through `kubernetes_manifest`.
- **Why** — keep workload definitions declarative and portable: the same module
  works on any conformant cluster (GKE, EKS, AKS, kind, …). Cloud-specific
  behaviour is opt-in via the dedicated GKE modules.
- **How you consume it** — every module takes a single `list(object({...}))`
  input and creates one object per entry with `for_each`, so a single module
  call can manage many objects of the same kind. Inputs are fully typed and
  validated.

> Each module has its own README with a description, the full input/output
> reference (generated by [terraform-docs](https://terraform-docs.io/)) and a
> usage example. Click any module in the catalog below.

## Usage

Reference a module by its Git source and a pinned ref, then pass a list of
objects:

```hcl
module "namespaces" {
  source = "github.com/YpNo/terraform-kubernetes-objects//modules/kubernetes-objects/namespace_v1?ref=v0.1.0"

  namespaces = [
    {
      name   = "team-a"
      labels = { "team" = "a" }
    },
    {
      name = "team-b"
    },
  ]
}
```

The modules manage *objects*, not the cluster connection — configure the
`kubernetes` provider in your root module:

```hcl
provider "kubernetes" {
  host                   = data.google_container_cluster.this.endpoint
  cluster_ca_certificate = base64decode(data.google_container_cluster.this.master_auth[0].cluster_ca_certificate)
  token                  = data.google_client_config.this.access_token
}
```

> **CRD modules (Istio / GKE / Gateway API):** these render through
> `kubernetes_manifest`, which contacts the API server **at plan time** and
> requires the target CustomResourceDefinition to already exist. Install the
> relevant CRDs in an earlier, separate apply. See [Notes](#notes).

## Testing

Every module ships a [`terraform test`](https://developer.hashicorp.com/terraform/language/tests)
suite under `tests/` that runs against a **mocked** provider (no live cluster
required):

```sh
cd modules/<group>/<module>
terraform init -backend=false
terraform test
```

## CI/CD

`.github/workflows/ci.yml` runs on every pull request and on pushes to `main`:

| Job             | What it does                                              |
| --------------- | --------------------------------------------------------- |
| Format          | `terraform fmt -check -recursive`                         |
| TFLint          | `tflint --recursive`                                      |
| Validate & Test | `terraform validate` + `terraform test` for every module  |
| tfsec           | static security analysis ([tfsec](https://github.com/aquasecurity/tfsec)) |
| Checkov         | policy-as-code scan ([Checkov](https://www.checkov.io/))  |

The same checks run locally via [pre-commit](https://pre-commit.com/)
(`.pre-commit-config.yaml`): `terraform fmt`, `terraform validate`, `tflint`.

## Releasing

Releases are automated with [release-it](https://github.com/release-it/release-it)
and driven by [Conventional Commits](https://www.conventionalcommits.org):

- **Pull requests** run the full CI suite.
- **On merge to `main`**, `.github/workflows/release.yml` computes the next
  semantic version from the commit history, updates `CHANGELOG.md`, creates the
  Git tag and a GitHub Release.

Consumers pin a module to a released tag via `?ref=vX.Y.Z` in the source URL.

## Notes

### CRD-based modules (Istio, GKE, Gateway API)

The Istio, GKE, and Gateway API modules render their objects through the
`kubernetes_manifest` resource, which has two constraints to plan around:

- **Plan-time API access** — `kubernetes_manifest` contacts the Kubernetes API
  server during `terraform plan`, not just `apply`. A reachable cluster is
  required even to plan.
- **CRDs must already be installed** — the target CustomResourceDefinition must
  exist on the cluster *before* the manifest is planned. Install the CRDs in a
  separate, earlier apply (or a distinct root module / `-target` run) ordered
  ahead of these modules.

### Versioned vs. legacy resource modules

Some Kubernetes object modules exist in both a legacy form (e.g. `configmap`)
and a versioned `_v1` form (e.g. `configmap_v1`). Prefer the `_v1` modules. The
non-versioned variants are deprecated aliases kept for backward compatibility
and will be removed in a future major release.
